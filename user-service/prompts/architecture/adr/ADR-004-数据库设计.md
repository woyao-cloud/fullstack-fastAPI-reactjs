# ADR-004: 数据库设计

## 状态
**已接受** (Accepted)

## 日期
2026-03-24

## 作者
系统架构师

---

## 上下文 (Context)

### 问题背景
系统需要设计数据库架构，满足以下需求：

- 支持1000万+用户规模
- 支持树形部门结构（最多5级）
- 支持RBAC四级权限模型
- 审计日志量大（1000万条/天）
- 高并发读写（登录10000 TPS）

### 约束条件
根据 CONTEXT.md 中的 **Locked Decisions**：
- D-02: 采用PostgreSQL 15作为关系型数据库
- D-07: 系统必须支持1000万+注册用户

### 数据库设计决策

| 设计点 | 决策 | 理由 |
|--------|------|------|
| **数据库** | PostgreSQL 15 | JSONB支持，性能优秀，Alembic兼容 |
| **主键** | UUID v4 | 分布式友好，安全（不可预测） |
| **软删除** | deleted_at字段 | 保留审计数据，支持恢复 |
| **时间戳** | created_at, updated_at | 审计追踪，数据同步 |
| **审计日志** | 按月分区 | 大数据量场景优化 |
| **部门树** | Materialized Path | 高效查询子树，支持5级层级 |
| **乐观锁** | version字段 | 并发更新保护 |

## 核心表设计

### 1. 用户表 (user)

```sql
CREATE TABLE user (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    email VARCHAR(255) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    first_name VARCHAR(100) NOT NULL,
    last_name VARCHAR(100) NOT NULL,
    phone VARCHAR(20),
    avatar_url VARCHAR(500),
    department_id UUID REFERENCES department(id),
    status VARCHAR(20) NOT NULL DEFAULT 'PENDING', -- ACTIVE, INACTIVE, PENDING, LOCKED
    email_verified BOOLEAN DEFAULT FALSE,
    failed_login_attempts INT DEFAULT 0,
    locked_until TIMESTAMP,
    last_login_at TIMESTAMP,
    last_login_ip VARCHAR(45),
    password_changed_at TIMESTAMP,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    deleted_at TIMESTAMP,
    version INT DEFAULT 0, -- 乐观锁

    CONSTRAINT chk_email_format CHECK (email ~* '^[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}$'),
    CONSTRAINT chk_phone_format CHECK (phone IS NULL OR phone ~ '^1[3-9]\d{9}$')
);

-- 索引
CREATE INDEX idx_user_email ON user(email);
CREATE INDEX idx_user_department ON user(department_id);
CREATE INDEX idx_user_status ON user(status);
CREATE INDEX idx_user_created ON user(created_at);
CREATE INDEX idx_user_deleted ON user(deleted_at) WHERE deleted_at IS NULL;
```

### 2. 部门表 (department)

```sql
CREATE TABLE department (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name VARCHAR(100) NOT NULL,
    code VARCHAR(50) UNIQUE NOT NULL, -- DEPT-001格式
    parent_id UUID REFERENCES department(id),
    manager_id UUID REFERENCES user(id),
    level INT NOT NULL CHECK (level BETWEEN 1 AND 5), -- 1=公司, 2=一级部门, ...
    path VARCHAR(500) NOT NULL, -- /1/2/5/10 材质化路径
    sort_order INT DEFAULT 0,
    description TEXT,
    status VARCHAR(20) DEFAULT 'ACTIVE', -- ACTIVE, INACTIVE
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    deleted_at TIMESTAMP,

    CONSTRAINT chk_no_self_parent CHECK (parent_id IS NULL OR parent_id != id),
    CONSTRAINT chk_no_circular_parent CHECK (parent_id IS NULL OR level > 1)
);

-- 索引
CREATE INDEX idx_dept_path ON department(path);
CREATE INDEX idx_dept_parent ON department(parent_id);
CREATE INDEX idx_dept_code ON department(code);
CREATE INDEX idx_dept_level ON department(level);
```

### 3. 角色表 (role)

```sql
CREATE TABLE role (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name VARCHAR(50) UNIQUE NOT NULL,
    code VARCHAR(50) UNIQUE NOT NULL, -- ROLE_ADMIN格式
    description TEXT,
    data_scope VARCHAR(20) NOT NULL DEFAULT 'ALL', -- ALL, DEPT, SELF, CUSTOM
    status VARCHAR(20) DEFAULT 'ACTIVE',
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    deleted_at TIMESTAMP
);

-- 索引
CREATE INDEX idx_role_code ON role(code);
```

### 4. 权限表 (permission)

```sql
CREATE TABLE permission (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name VARCHAR(100) NOT NULL,
    code VARCHAR(100) UNIQUE NOT NULL, -- user:create格式
    type VARCHAR(20) NOT NULL, -- MENU, ACTION, FIELD, DATA
    resource VARCHAR(50) NOT NULL, -- user
    action VARCHAR(50), -- create, update, delete, read
    parent_id UUID REFERENCES permission(id),
    icon VARCHAR(100), -- 菜单图标
    route VARCHAR(200), -- 前端路由
    sort_order INT DEFAULT 0,
    status VARCHAR(20) DEFAULT 'ACTIVE',
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

-- 索引
CREATE INDEX idx_perm_code ON permission(code);
CREATE INDEX idx_perm_type ON permission(type);
CREATE INDEX idx_perm_parent ON permission(parent_id);
```

### 5. 角色-权限关联表 (role_permission)

```sql
CREATE TABLE role_permission (
    role_id UUID NOT NULL REFERENCES role(id),
    permission_id UUID NOT NULL REFERENCES permission(id),
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,

    PRIMARY KEY (role_id, permission_id)
);

-- 索引
CREATE INDEX idx_rp_permission ON role_permission(permission_id);
```

### 6. 用户-角色关联表 (user_role)

```sql
CREATE TABLE user_role (
    user_id UUID NOT NULL REFERENCES user(id),
    role_id UUID NOT NULL REFERENCES role(id),
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,

    PRIMARY KEY (user_id, role_id)
);

-- 索引
CREATE INDEX idx_ur_role ON user_role(role_id);
```

### 7. 审计日志表 (audit_log) - 分区表

```sql
-- 主表（模板）
CREATE TABLE audit_log (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID REFERENCES user(id),
    username VARCHAR(100), -- 冗余存储，用户可能已删除
    operation VARCHAR(50) NOT NULL, -- CREATE, UPDATE, DELETE, LOGIN, etc.
    resource_type VARCHAR(50) NOT NULL, -- USER, ROLE, PERMISSION, etc.
    resource_id UUID,
    old_value JSONB,
    new_value JSONB,
    description TEXT,
    client_ip VARCHAR(45),
    user_agent TEXT,
    session_id VARCHAR(100),
    success BOOLEAN NOT NULL,
    error_message TEXT,
    execution_time_ms INT, -- 执行耗时
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
) PARTITION BY RANGE (created_at);

-- 创建分区（按月）
CREATE TABLE audit_log_2026_03 PARTITION OF audit_log
    FOR VALUES FROM ('2026-03-01') TO ('2026-04-01');

CREATE TABLE audit_log_2026_04 PARTITION OF audit_log
    FOR VALUES FROM ('2026-04-01') TO ('2026-05-01');

-- 索引
CREATE INDEX idx_audit_user ON audit_log(user_id);
CREATE INDEX idx_audit_time ON audit_log(created_at);
CREATE INDEX idx_audit_operation ON audit_log(operation);
CREATE INDEX idx_audit_resource ON audit_log(resource_type, resource_id);
```

## 备选方案 (Alternatives)

### 备选1: 使用自增ID作为主键

**优点：**
- 存储空间小
- 查询性能略好
- 可读性高

**缺点：**
- 分布式场景ID冲突
- 可预测性高（安全隐患）
- 数据迁移复杂

**决策：** 拒绝。使用UUID更适合分布式架构。

### 备选2: 审计日志使用独立数据库

**优点：**
- 隔离业务数据和日志数据
- 日志数据库可独立优化

**缺点：**
- 增加运维复杂度
- 跨库事务困难
- 初期数据量不需要

**决策：** 拒绝。使用分区表足够，后期可考虑归档到ClickHouse。

### 备选3: 使用嵌套集模型存储部门树

**优点：**
- 查询子树无需递归

**缺点：**
- 插入更新复杂度高（需要更新多个节点）
- 不支持无限层级（性能下降）

**决策：** 拒绝。使用Materialized Path更平衡，更新简单，查询高效。

---

## 后果 (Consequences)

### 正面后果

1. **可扩展性**：分区表支持大数据量审计日志
2. **安全性**：UUID主键不可预测，防止ID遍历攻击
3. **可维护性**：软删除保留历史数据，支持审计
4. **性能**：合理索引设计支持高并发查询
5. **灵活性**：JSONB字段支持扩展字段

### 负面后果

1. **存储空间**：UUID比自增ID占用更多空间
2. **查询性能**：UUID索引比整数索引略慢
3. **可读性**：UUID不如数字ID直观
