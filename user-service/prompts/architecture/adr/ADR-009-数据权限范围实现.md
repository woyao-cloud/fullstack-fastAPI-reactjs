# ADR-009: 数据权限范围实现

## 状态
Accepted

## 背景
根据FRD v1.2需求，系统需要支持细粒度的数据权限控制。除了菜单、操作、字段权限外，还需要支持数据行级别的权限控制。具体需要实现四种数据权限范围：全部(ALL)、本部门(DEPT)、本人(SELF)、自定义(CUSTOM)。

## 决策
采用 **角色级数据范围配置** + **查询时动态过滤** 的方式实现数据权限控制。

## 架构设计

### 1. 数据模型设计

```sql
-- 角色表增加数据范围字段
ALTER TABLE role ADD COLUMN data_scope VARCHAR(20) NOT NULL DEFAULT 'SELF';
ALTER TABLE role ADD COLUMN data_conditions JSONB;

-- 数据范围枚举
-- ALL: 全部数据
-- DEPT: 本部门及子部门数据
-- SELF: 本人创建的数据
-- CUSTOM: 自定义条件
```

### 2. 权限检查流程

```
用户请求 → JWT认证 → 获取用户角色 → 查询数据范围配置
                                         ↓
                               根据data_scope生成过滤条件
                                         ↓
                               应用到查询语句
                                         ↓
                               返回过滤后的数据
```

### 3. 实现方案选择

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **方案A: 数据库视图** | 为每个角色创建数据库视图 | 性能好，数据库级安全 | 动态创建视图复杂，维护困难 |
| **方案B: 应用层过滤** | 在应用层添加WHERE条件 | 灵活，易于实现和维护 | 性能依赖应用层实现 |
| **方案C: 行级安全** | 使用PostgreSQL行级安全策略 | 数据库级安全，性能好 | 配置复杂，跨数据库不兼容 |
| **方案D: 数据标记** | 每条数据标记可见范围 | 简单直观 | 数据冗余，更新复杂 |

**选择方案B（应用层过滤）的原因**：
1. **灵活性高**：支持复杂的自定义条件
2. **易于维护**：代码控制，无需数据库特殊配置
3. **跨数据库兼容**：不依赖特定数据库特性
4. **团队熟悉**：SQLAlchemy 查询构造 + FastAPI 依赖注入

## 具体实现

### 1. 数据范围枚举

```python
from enum import Enum

class DataScope(str, Enum):
    ALL = "ALL"        # 全部数据
    DEPT = "DEPT"      # 本部门及子部门数据
    SELF = "SELF"      # 本人数据
    CUSTOM = "CUSTOM"  # 自定义范围
```

### 2. 数据权限过滤依赖

```python
from fastapi import Depends
from sqlalchemy import select
from sqlalchemy.ext.asyncio import AsyncSession

class DataPermissionFilter:
    def __init__(self, department_service: DepartmentService):
        self.department_service = department_service

    async def apply_filter(
        self,
        stmt: select,
        user: CurrentUser,
        data_scope: DataScope,
        custom_conditions: str | None,
    ) -> select:
        """根据数据范围生成过滤条件并叠加到查询语句"""
        if data_scope is DataScope.ALL:
            return stmt  # 无过滤

        if data_scope is DataScope.DEPT:
            return await self._apply_department_filter(stmt, user.department_id)

        if data_scope is DataScope.SELF:
            return stmt.where(model.created_by == user.id)

        if data_scope is DataScope.CUSTOM:
            return await self._parse_custom_conditions(stmt, custom_conditions, user)

        raise ValueError(f"未知的数据范围: {data_scope}")

    async def _apply_department_filter(self, stmt, department_id: UUID) -> select:
        # 获取部门子树ID列表
        accessible_dept_ids = await self.department_service.get_sub_department_ids(department_id)
        accessible_dept_ids.append(department_id)
        return stmt.where(model.department_id.in_(accessible_dept_ids))

    async def _parse_custom_conditions(self, stmt, conditions: str, user: CurrentUser) -> select:
        # 解析JSON格式的自定义条件
        # 例如: {"departmentId": "dept123", "status": "ACTIVE"}
        # 支持变量替换，如 ${userId}, ${departmentId}
        return await DataConditionParser.parse(stmt, conditions, user)
```

> 实现说明：在 FastAPI 中通过依赖注入获取当前用户与数据库会话，在 Service/Repository 层对 SQLAlchemy `select` 语句叠加 `where` 条件完成数据过滤；亦可封装为 FastAPI middleware 在请求级统一注入过滤上下文。


## 后果

### 正面影响
1. **细粒度控制**：支持四种数据范围，满足企业级需求
2. **灵活性高**：自定义条件支持复杂业务场景
3. **易于扩展**：可以轻松添加新的数据范围类型
4. **性能可控**：通过缓存和优化减少性能影响

### 负面影响
1. **性能开销**：应用层过滤相比数据库级方案有额外开销
2. **实现复杂度**：需要处理各种边界情况和错误处理
3. **缓存一致性**：部门结构调整时需要更新相关缓存

### 缓解措施
1. **性能优化**：使用缓存，优化查询，批量处理
2. **代码质量**：编写单元测试，代码审查，文档完善
3. **监控告警**：监控数据权限查询性能，设置阈值告警

## 替代方案

### 方案A：数据库行级安全（RLS）
```sql
-- PostgreSQL行级安全策略
ALTER TABLE user ENABLE ROW LEVEL SECURITY;
CREATE POLICY user_access_policy ON user
    USING (department_id IN (
        SELECT get_accessible_departments(current_user)
    ));
```
**放弃原因**：配置复杂，跨数据库不兼容，团队不熟悉。

### 方案B：数据标记法
```sql
-- 每条数据标记可见范围
ALTER TABLE user ADD COLUMN visibility VARCHAR(20);
-- 值: PUBLIC, DEPARTMENT, PRIVATE
```
**放弃原因**：数据冗余，更新复杂，不支持动态条件。
